## Session Update: 2026-06-19 03:25 PDT — SPI Display Debug

### Current Problem
ILI9341 display not responding to `spi disp id` command. Returns `0x00 0x00 0x00` (not expected `0x00 0x93 0x41`).

### Root Cause Analysis
**GPIO direction lock issue:** `IO_GPIO_DIR_EN=false` in BD means GPIO output directions are locked at synthesis time. GPIO DIR register `0x00000048` = only bits 3 (RSTCN) and 6 (CLK_EN) are outputs. All other GPIOs are inputs.

**Affected:** GPIO5 (DC pin for display) is INPUT — can't drive DC LOW/HIGH from firmware. ILI9341 sees DC=LOW for all SPI bytes, misinterprets read data as commands.

### GPIO Map (basic_plus_top, LOCKED by BD)
| Bit | Pin | Function | Direction | Status |
|-----|-----|----------|-----------|--------|
| 0 | — | LED | Output | ✅ Works (driven by LED pin) |
| 1 | E13 | a0 (I²C addr) | Input | ❌ Can't drive from FW |
| 2 | E14 | a1 (I²C addr) | Input | ❌ Can't drive from FW |
| 3 | D14 | RSTCN (codec) | **Output** | ✅ Works |
| 4 | T5 | RSTDN (display) | Input | ❌ Can't drive from FW |
| 5 | U5 | DC (display) | Input | ❌ Can't drive from FW |
| 6 | E13 | CLK_EN | **Output** | ✅ Works |

### SPI CS Lines (all working — firmware uses CSEN via command-mode FIFO entries)
- CS0 (V5) = display ILI9341
- CS1 (U6) = touch XPT2046
- CS2 (V9) = SD card
- CS3 (V8) = flash W25Q64

### Workaround Options (in priority order)
1. **Re-synth BD** with GPIO5 (DC) and GPIO4 (RSTDN) as outputs — proper fix
2. **Use NeoRV32 SPI command mode** to handle CS+DC timing in one FIFO entry
3. **Bit-bang DC via PWM output** if PWM channel can be routed to the DC pin (not possible without re-synth)

### What Works Right Now
- `spi flash id` → CS3 works, returns `0x00 0x00 0x00` (flash not soldered, MISO floats low)
- `spi disp id` → CS0 works (LA confirms CS activates), but display in reset state returns 0x00 for all bytes
- `codec init` → CS42448 at I²C 0x48, all 7 registers written+verified ✅
- `pwm init/set N/off` → display backlight control works on led_pwm pin (T4)
- `i2c scan/probe/read/write` → full I²C bus control ✅

### Immediate Next Steps (after re-synth BD)
1. Rebuild bitstream with `IO_GPIO_DIR_EN=true` OR set GPIO4/GPIO5 as outputs explicitly
2. Reflash debug_console firmware
3. Test `spi disp id` — should return `0x00 0x93 0x41` if display has power
4. Add ILI9341 full init sequence (hardware reset → soft reset → sleep out → display on → read ID)

### Lessons Learned This Session (add to HARDWARE-DEBUG-RULES.md when updated)
- **When `IO_GPIO_DIR_EN=false`, `gpio dump` readback is UNRELIABLE** — check via OpenOCD `mdw 0xFFFC0000` for actual OUTPUT register
- **SPI CS lines work via CSEN in command-mode FIFO entries** — the `neorv32_spi_cs_en()` + `neorv32_spi_transfer()` sequence DOES assert CS (verified by LA)
- **Return value of 0x00 from SPI ≠ CS failure** — device may be in reset, unpowered, or responding to wrong command (DC pin issue)
- **Always check LA BEFORE assuming hardware works** — CS activation visible on LA is ground truth, firmware reports are not
## GPIO Architecture Lesson (v29 — 2026-06-19 04:20 PDT)

### THE BIG MISTAKE
NeoRV32 GPIO has TWO separate registers:
- **PORT_IN @ GPIO_BASE+0x00** — read-only, reads `gpio_i` pins from fabric
- **PORT_OUT @ GPIO_BASE+0x04** — read/write, drives `gpio_o` pins to fabric

`neorv32_gpio_pin_get()` reads PORT_IN. In this build, ONLY `int_in` (touch PENIRQ, R4) is wired to `gpio_i`. So `gpio dump` was showing garbage — it was reading the input register, not the output pins we drove.

**Fix:** Added `gpio_out_get()` (reads PORT_OUT) and `gpio_in_get()` (reads PORT_IN). `gpio dump` now shows both registers separately. `gpio r` reads PORT_OUT. `gpio in` reads PORT_IN.

### CORRECTED GPIO OUTPUT PIN MAP (from block design + XDC)
| Bit | Name | Pin | Function |
|-----|------|-----|----------|
| 0 | LED | G2 | On-board LED, active HIGH |
| 1 | a0 | E13 | CS42448 I²C address LSB |
| 2 | a1 | E14 | CS42448 I²C address MSB |
| 3 | RSTCN | D14 | Codec reset, active HIGH |
| 4 | RSTDN | T5 | Display reset, active HIGH |
| 5 | DC | U5 | Display data/command |
| 6 | CLK_EN | (internal) | Clock gen enable, HIGH=running |

### CORRECTED SPI CS MAP (from impl_1 io_placed.rpt)
| CS Index | spi_cs bit | Pin | Device |
|----------|-----------|-----|--------|
| CS0 | spi_cs[0] | V5 | W25Q64 flash |
| CS1 | spi_cs[1] | U6 | SD card |
| CS2 | spi_cs[2] | V9 | ILI9341 display |
| CS3 | spi_cs[3] | V8 | XPT2046 touch |

**Source of truth for CS mapping:** Vivado impl_1 `neorv_basic_plus_top_io_placed.rpt` — NOT the stale XDC which uses old port names.

### LED Blink
LED (GPIO bit 0) blinks every ~2s in the CLI loop. Steady during bootloader.

### Backlight
PWM ch0 initialized at 50% duty (127/255) on startup.

### SPI CS Bug History
The firmware had CS indices scrambled — display on CS2, touch on CS3, SD on CS1, flash on CS0. The XDC names (t_spi_cssd, t_spi_cstch, t_spi_csdsp, t_spi_csfl) don't match the current top module's `spi_cs[3:0]` bus. The actual mapping was determined from the placed I/O report.

## v29 Stable — Canonical Debug Process (2026-06-19 04:57 PDT)

### Flash Method (PROVEN — do not deviate)
1. Build: `cd firmware/debug_console && make clean && make exe`
2. Flash: `sleep 3 && JLinkExe -device RISC-V -if JTAG -speed 1000 -autoconnect 1 -JTAGConf "-1,-1" -CommanderScript flash.jlink`
   - flash.jlink: r → halt → loadbin elf.bin 0x00000000 → wreg pc 0 → go → exit
   - MUST use elf.bin (raw binary), NOT neorv32_exe.bin
   - MUST use 1000 kHz JTAG speed (higher = intermittent failures)
   - MUST wait 3-5s before connecting (target needs recovery time)
   - DO NOT retry rapidly — each reset destabilizes the TAP
3. UART: `/dev/ttyUSB0` @ 19200 baud, Python serial, keep connection open

### GPIO Architecture (DO NOT FORGET)
- PORT_IN @ 0xFFFC0000+0x00 = gpio_i pins from fabric (read-only)
- PORT_OUT @ 0xFFFC0000+0x04 = gpio_o pins to fabric (read/write)
- NO DIR register (IO_GPIO_DIR_EN=false) — direction is fixed by port wiring
- gpio_o bits: 0=LED(G2) 1=a0(E13) 2=a1(E14) 3=RSTCN(D14) 4=RSTDN(T5) 5=DC(U5) 6=CLK_EN(int)
- gpio_i bits: only bit 0 = int_in(R4) = touch PENIRQ
- neorv32_gpio_pin_get() reads PORT_IN — useless for output pins!
- Use direct register reads: NEORV32_GPIO->PORT_OUT for outputs, ->PORT_IN for inputs

### SPI CS Map (from placed I/O report, not stale XDC)
- CS0 = flash (V5, spi_cs[0])
- CS1 = SD card (U6, spi_cs[1])
- CS2 = display (V9, spi_cs[2])
- CS3 = touch (V8, spi_cs[3])

### Display Status

- Backlight 50% at boot (PWM ch0, 127/255)
- ID read returns 0x00 0x00 0x00 — hardware issue, not firmware

- Does NOT interfere with UART (no delay_ms in input loop)
