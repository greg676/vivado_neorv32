## CRITICAL LESSONS LEARNED (today's battle scars)

### 1. RP2040 Passthrough: Thonny/MicroPython is fragile
- **DO NOT use auto-run `main.py`** — `sys.stdin` blocks the VM and kills REPL
- **DO use Thonny "Run current script"** — keeps REPL alive underneath
- CircuitPython = extra pain (`boot.py`, dual CDC ports, `board.LED` doesn't exist)
- Arduino C firmware would be more reliable for dedicated passthrough (non-blocking Serial↔Serial1)

### 2. J-Link loadfile does a RESET
- `loadfile` → implicit reset → bootloader restarts → PC goes to 0xFFE00000, NOT our code
- **Correct sequence:** `loadfile` (to get code into IMEM) then `wreg pc, 0` then `go`
- This works because IMEM = block RAM, not flash — it persists across resets
- `loadbin` also resets. Both require post-load PC override.

### 3. IMEM is BRAM, not Flash
- No SPI flash chip on the board → bootloader errors on SPI flash probe
- IMEM at 0x00000000 is FPGA block RAM initialized by the bitstream
- J-Link loads code into BRAM; it persists until next FPGA reconfiguration
- Bootloader doesn't auto-execute from IMEM unless correctly formatted EXE image

### 4. Clock mismatch drives EVERYTHING
- CLOCK_FREQUENCY generic = 100MHz but actual MMCM output = 160MHz
- **UART baud = 19200 × 1.6 = 30720** — had to discover this experimentally
- MUST fix CLOCK_FREQUENCY generic in Vivado IP before next build

### 5. Toolchain: Vitis GCC works, but needs wrappers
- System riscv64-unknown-elf-gcc has NO newlib — can't compile bare-metal
- Vitis `riscv32-xilinx-elf-gcc` has full newlib with sysroot
- Created wrapper scripts (`tools/riscv32-unknown-elf-*`) that point to Vitis binaries
- Makefile uses `RISCV_PREFIX ?= riscv32-unknown-elf-`

### 6. Build output location
- `make exe` outputs `neorv32_exe.bin` and `main.elf` to PROJECT ROOT, not `build/`
